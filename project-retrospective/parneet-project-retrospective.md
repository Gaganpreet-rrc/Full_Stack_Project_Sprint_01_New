# Project Retrospective – Parneet Kaur

## When was the team most effective?

- Our team was most effective when tasks were clearly divided among members. For example, some worked on frontend, others on backend, which made development faster and more organized.  
- We worked well when using Git branches properly, as it allowed everyone to work independently without breaking the main code.  
- Communication through messages and updates helped us solve issues quickly, especially when connecting frontend and backend features like Library Tips.  
- The team was also efficient when we focused on one feature at a time and tested it before moving to the next step.  

---

## When was the team least effective?

- The team was least effective during integration stages, especially when connecting backend, database, and authentication. There were many errors that took time to debug.  
- We sometimes faced merge conflicts because multiple people were working on similar files, which slowed down progress.  
- At times, we did not test features step by step, which caused multiple issues to appear at once and made debugging more difficult.  
- There was some confusion while working with new tools like Clerk authentication and API handling, which affected efficiency.  

---

## What did I learn for future work?

- I learned that breaking work into smaller steps and testing each part (backend, API, frontend) separately helps reduce errors and saves time.  
- I will continue using Git branches and make frequent commits with clear messages to keep my work organized.  
- In the future, I will spend more time understanding new technologies (like authentication systems and APIs) before starting implementation.  
- I want to avoid working on multiple parts at the same time and instead focus on completing one part fully before moving on.  
- I will improve my debugging skills by carefully reading errors and checking the exact source of the problem instead of guessing.
